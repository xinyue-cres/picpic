# picpic — 本地照片清理工具 设计文档

日期:2026-07-06
状态:已确认,待进入实现计划

---

## 1. 背景与目标

用户安卓手机存量照片约 2w 张,手动归类整理极耗时。希望有一个**本地运行**的工具,自动识别哪些照片可以删除,辅助快速清理。

**核心诉求**
- 主要想删两类:**无价值内容**、**截图/非拍摄类**;分类标准**用户以后能自己扩展**。
- 删除必须**安全可逆**:程序挑出"待删候选",用户看图确认后才批量删,且删除是移入回收区而非真删。
- **全本地运行**,照片不上网(隐私)。
- 用户能接受**命令行跑批 + 网页界面看图确认**。

**运行环境**:照片从安卓手机手动导出成文件夹后,在电脑上处理(Windows PC 或 Mac 均可)。不在安卓手机上直接跑 ML(太重)。

**非目标(YAGNI)**
- 不做长期相册管理(不与 immich/PhotoPrism 竞争)。
- 不做手机端 App。
- 不自动删除任何文件——最终删除动作永远由用户在界面上确认。
- 不追求语义分类 100% 准确;CLIP 只产出候选,用户目视兜底。

---

## 2. 现状调研(为什么自建)

现成开源工具只覆盖一半需求:

| 工具 | 能做 | 缺口 |
|---|---|---|
| Czkawka | 去重、找相似、模糊废片,成熟,有 GUI/CLI | 不懂"截图 vs 拍摄"、不懂"无价值内容"、不能自定义语义分类 |
| immich / PhotoPrism | AI 打标签、物体/场景识别、以图搜图 | 要架服务器,重;面向"管理相册"非"挑出来删";规则不好当删除规则用 |
| digiKam | 重量级照片管理 + 部分 AI 标签 | 笨重,交互不面向批量清理 |

**结论**:重复/模糊这类机械判断,成熟库直接用,不重造轮子;但"截图/非拍摄""无价值内容"+"分类标准可自定义"现成工具做不好,需用**本地 CLIP 零样本分类**补上——加一个文字类别即多识别一类图,天然满足"以后自己界定"。

## 3. 整体架构与数据流

**核心原则:原图只读,永不原地修改。** 所有判断结果写入本地 SQLite,删除 = 移动到回收区,可还原。

```
安卓手机
   │  (用户手动导出,USB/无线,一次性)
   ▼
照片文件夹 (只读)
   │
   ▼
┌─────────────────────────────────────────┐
│  picpic 流水线 (Python CLI)               │
│  1. scan    遍历文件夹 → 登记到 SQLite     │
│  2. analyze 跑检测器 → 分数写回 SQLite     │
│       ├ EXIF 判截图                        │
│       ├ 感知哈希 (相似分组)                │
│       ├ 模糊检测                           │
│       └ CLIP 语义分类 (阶段2)              │
│  3. rules   按规则算出 verdict 标记         │
└─────────────────────────────────────────┘
   │
   ▼
SQLite (picpic.db) ── 唯一真相源,所有元数据/分数/标记
   │
   ▼
┌─────────────────────────────────────────┐
│  网页确认界面 (本地服务, 浏览器打开)        │
│   - 待删候选区:平铺网格,多选确认           │
│   - 相似图组审查区:按组并排,组内挑删       │
│   - 回收区:还原 / 清空(真删)              │
└─────────────────────────────────────────┘
   │
   ▼
回收区文件夹 (_picpic_trash/)  ← 移动,不真删,可还原
```

**关键设计**
- 全本地,照片不上网。
- `scan` / `analyze` / `rules` / `界面` 是**四个独立步骤**,各自可单独重跑。改规则只需重跑 `rules`,不必重新分析 2w 张。
- 数据库是唯一真相源,原图和界面都围绕它。

## 4. 数据模型 (SQLite)

一张核心表 `photos`,一行 = 一张图:

```
photos
├─ id                主键
├─ path              原图绝对路径 (只读,永不改)
├─ file_hash         文件内容哈希 (完全相同的文件)
├─ phash             感知哈希 (视觉相似分组用)
├─ width, height     尺寸
├─ file_size         文件大小
├─ created_at        拍摄/创建时间 (从 EXIF 读)
├─ camera_model      相机型号 (有=拍的, 无=可能截图/网图)
│
├─ is_screenshot     ← 检测器填:是否截图
├─ blur_score        ← 检测器填:模糊程度 (越低越糊)
├─ dup_group         ← 检测器填:相似图分到同一组号
├─ clip_labels       ← 检测器填(阶段2):CLIP 语义标签+分数, JSON
│
├─ verdict           规则结论:keep / trash_candidate
├─ verdict_reason    原因 (screenshot / blurry / dup / ...)
├─ status            当前状态:active / trashed
└─ trashed_at        移入回收区时间 (还原用)
```

**设计考量**
- **检测结果**(客观事实)与**最终结论**(`verdict`,规则算出)分离存储。改规则只重算 `verdict`,不重跑检测——这是"加规则很便宜"的底层原因。
- `verdict_reason` 让界面显示"为什么被挑出",加快用户判断(用户明确要求保留)。
- `status` + 回收区实现**可还原**。
- 未来加新分类维度(如"是否收据"),加列或塞进 `clip_labels` JSON,不动其他逻辑。

---

## 5. 检测器

按"先便宜后贵"排序。前四个不依赖 CLIP(阶段1),最后一个是 CLIP(阶段2)。

| 检测器 | 判断方式 | 库 | 准度 |
|---|---|---|---|
| 完全重复 | 文件内容哈希一致 = 同一文件 | `hashlib` | 100% |
| 视觉相似 | 感知哈希 (phash) 距离近 → 归一组 | `imagehash` | 高,连拍/改尺寸可抓 |
| 模糊废片 | 拉普拉斯方差,越低越糊 | `opencv` | 高,阈值需微调 |
| 截图/非拍摄 | EXIF 无相机型号 + 常见屏幕分辨率 | `Pillow` | 较高,少量误判靠界面兜底 |
| 语义分类(阶段2) | CLIP 拿文字类别零样本打分 | `open_clip` | 候选级 |

**已确认的关键决定**
1. **相似图不自动取舍**:程序只把视觉相似的圈到同一 `dup_group`,**不自动决定保留哪张**。整组拎到界面"相似图组审查区",用户组内手动挑删。
2. **模糊阈值不写死**:给保守默认值,`analyze` 后在界面用滑块实时调,配合"模糊排行榜"目视定线。
3. **截图判断含网图**:主要依据"EXIF 无相机型号",会把从微信/网上存的图也归入非拍摄类——符合用户要删的预期。

## 6. 网页确认界面

`analyze` 完成后启动本地服务(浏览器自动打开 localhost),纯本地不上网。

**布局:顶部页签切换三个场景**

```
┌───────────────────────────────────────────────┐
│  [待删候选]  [相似图组]  [回收区]      共 2w 张   │
├───────────────────────────────────────────────┤
│  筛选: ☑截图 ☑模糊 ☑重复   模糊阈值 ▓▓▓░░ [滑块] │
├───────────────────────────────────────────────┤
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐  ← 缩略图网格          │
│  │☑ │ │☑ │ │  │ │☑ │ │  │    每张角标显示理由     │
│  │截│ │糊│ │  │ │重│ │  │   (截图/糊/重复)        │
│  └──┘ └──┘ └──┘ └──┘ └──┘                         │
├───────────────────────────────────────────────┤
│  已选 1,203 张   [移入回收区]  [全选] [取消]       │
└───────────────────────────────────────────────┘
```

**交互要点**
- **缩略图**:用生成的小图渲染(2w 张原图直出会卡死);点击放大看原图。
- **理由角标**:每张显示 `verdict_reason`,一眼知道为何被挑出。
- **筛选 + 滑块**:实时调"模糊到什么程度算废",网格随之更新。
- **相似图组页**:按 `dup_group` 分块,一组并排,组内勾选删除。
- **移入回收区**:选中项**移动**到 `_picpic_trash/`,DB 标 `trashed`,**不真删**。
- **回收区页**:查看已删、**还原**、"清空回收区"(此步才真删,单独二次确认)。

**技术选型**:后端 Python(FastAPI 起本地服务读 SQLite),前端单页面(原生 JS 或极简框架,不搞复杂前端工程)。轻量、够用、易改。

## 7. 规则引擎

`verdict` 由规则算出,规则是**数据/配置**而非硬编码,可随时增删。

示例规则(概念形态):
- `is_screenshot == true` → `trash_candidate`,reason=`screenshot`
- `blur_score < 阈值` → `trash_candidate`,reason=`blurry`
- `file_hash 有完全重复` → 除 1 张外其余 `trash_candidate`,reason=`exact_dup`
- 相似组(`dup_group`)**不自动判**,交界面人工处理

规则层独立于检测层:改规则 → 只重跑 `rules` 命令 → 重算 `verdict`。

---

## 8. 安全设计(删除不可逆的防护)

- 原图全程**只读**。
- "删除"= 移动到 `_picpic_trash/` + DB 标 `trashed`,**可一键还原**。
- 真正物理删除只发生在"清空回收区",且**单独二次确认**。
- 数据库与原图分离,重跑任何步骤都不损坏原图。

---

## 9. 分阶段交付

每阶段独立可用:

- **阶段 1(MVP)**:scan + 去重 + 模糊 + 截图判断 + 网页确认删除 + 回收区。**不依赖 CLIP**,即可清掉可能一大半图。
- **阶段 2**:加 CLIP 语义分类 + 用户自定义文字类别。
- **阶段 3**:规则引擎做成可配置文件,用户自由增删规则。

---

## 10. 风险与权衡

1. **CLIP 在 2w 张上的速度**:有无 GPU 差别大,纯 CPU 可能数小时(可后台过夜)。缓解:廉价信号先跑,CLIP 最后跑;阶段 1 完全不需要 CLIP。
2. **分类准确度**:截图/重复/模糊很准;"无价值内容"主观,CLIP 只给候选,靠界面目视兜底,不追求全自动准确。

- 语言:Python 3

- 语义(阶段2):open_clip

- 跨平台:Windows / Mac 均可本地运行
